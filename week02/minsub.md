# 오답노트

## 문제 1

### 문제 정보

- 문제명: 행렬 테두리 회전하기 
- 문제 링크: https://school.programmers.co.kr/learn/courses/30/lessons/77485
- 사용한 알고리즘 / 자료구조: 구현

### 처음 접근 방식

시계 방향으로 회전해서 테두리 숫자들 중 가장 작은 수를 저장하는 문제라고 생각했다.

그래서 먼저 `rows x columns` 크기의 2차원 배열을 만들고, 1부터 순서대로 값을 채웠다.  

이후 각 `query`마다 지정된 사각형의 테두리를 시계 방향으로 한 칸씩 회전시키고, 회전 과정에서 이동한 값들 중 최솟값을 구해 `answer`에 저장하는 방식으로 접근했다.

### 풀이 방법

각 query는 `[x1, y1, x2, y2]` 형태로 주어지지만, 배열 인덱스는 0부터 시작하므로 각각 1씩 빼서 `top`, `left`, `bottom`, `right`로 변환했다.

회전은 테두리를 네 방향으로 나누어 처리했다.

1. 위쪽 행: 왼쪽 → 오른쪽
2. 오른쪽 열: 위 → 아래
3. 아래쪽 행: 오른쪽 → 왼쪽
4. 왼쪽 열: 아래 → 위

이때 값을 덮어쓰면 기존 값이 사라지기 때문에, `previous` 변수에 이전 값을 저장해두고 현재 칸에 넣는 방식으로 회전시켰다.

처음에는 왼쪽 아래 방향에서 올라올 값인 `graph[top + 1][left]`를 `previous`로 저장했다.  
이후 각 방향을 순회하면서 현재 칸의 값을 `temp`에 저장하고, 현재 칸에는 `previous`를 넣은 뒤, `previous`를 `temp`로 갱신했다.

이 과정을 반복하면 테두리 값들이 시계 방향으로 한 칸씩 이동하게 된다.  
또한 회전 과정에서 `previous`에 담기는 값들을 기준으로 최솟값을 계속 갱신해, 해당 query의 결과로 저장했다.

### 어려웠던 지점

회전 로직을 구현할 때 각 가장자리 값들의 이동 방향이 달라지기 때문에, 반복문 범위를 어떻게 잡아야 할지가 어려웠다.

특히 값을 바로 덮어쓰면 원래 값이 사라지기 때문에, 이전 값을 임시로 저장하면서 이동시키는 방식이 필요했다.  

처음에는 각 방향마다 값을 따로 저장해야 하나 고민했지만, 왼쪽 위로 올라올 값을 `previous`로 미리 잡아두면 네 방향을 순서대로 순회하면서 자연스럽게 회전시킬 수 있다는 점을 떠올리는 것이 핵심이었다.

### 해결 방법

테두리 회전을 하나의 큰 작업으로 보지 않고, 네 개의 방향 이동으로 나누어 구현했다.



### 시간복잡도 / 공간복잡도

- 시간복잡도: O(rows * columns + 모든 query의 테두리 길이 합)
- 공간복잡도: O(rows * columns)

## 문제 3

### 문제 정보

- 문제명: 문자열 압축
- 문제 링크: https://school.programmers.co.kr/learn/courses/30/lessons/60057
- 사용한 알고리즘 / 자료구조: 문자열, 구현, 완전 탐색

### 처음 접근 방식


문자열을 1개 단위부터 전체 문자열 길이까지 잘라보면서, 같은 단위 문자열이 연속해서 몇 번 반복되는지 카운팅하면 된다고 생각했다.

각 단위 길이마다 압축 문자열을 만들어보고, 그중 가장 짧은 압축 문자열의 길이를 정답으로 구하는 방식으로 접근했다.

### 풀이 방법

문자열을 자르는 단위 길이를 `1`부터 `s.length()`까지 모두 시도했다.

각 단위 길이마다 첫 번째 문자열 조각을 `target`으로 잡고, 이후 문자열을 같은 길이만큼 잘라 `comp`와 비교했다.

- `comp`가 `target`과 같으면 같은 문자열이 반복된 것이므로 `count`를 증가시킨다.
- `comp`가 `target`과 다르면 지금까지 누적된 `target`을 압축 문자열에 추가한다.
    - `count >= 2`이면 숫자와 함께 추가한다.
    - `count == 1`이면 문자열만 추가한다.
- 이후 `target`을 `comp`로 바꾸고, `count`를 다시 1로 초기화한다.

반복문이 끝난 뒤에는 마지막으로 남아 있는 `target`도 압축 문자열에 추가해야 한다.  
각 단위 길이마다 만들어진 압축 문자열의 길이를 구하고, 그중 최솟값을 정답으로 갱신했다.

### 어려웠던 지점

문자열을 몇 글자 단위로 자를지 정하고, 그 단위에 맞춰 이전 문자열과 현재 문자열을 비교하는 구조를 떠올리는 것이 어려웠다.

특히 반복이 끝난 뒤 마지막 `target`은 비교 대상이 더 이상 없기 때문에, 반복문 밖에서 한 번 더 압축 문자열에 추가해야 한다는 점을 놓치기 쉬웠다.

또한 문자열 길이가 단위 길이로 딱 나누어떨어지지 않을 수 있기 때문에, `Math.min(j + i, s.length())`를 사용해 마지막 substring 범위를 처리해야 했다.


### 해결 방법
단위 길이를 기준으로 문자열을 잘라가며 이전 조각과 현재 조각을 비교했다.

반복이 끝난 뒤에는 마지막으로 남은 target을 처리했다.

```java
if (count >= 2) {
    word += count;
}

word += target;
```
이후 만들어진 압축 문자열의 길이와 기존 정답을 비교해 더 짧은 길이로 갱신했다.

### 시간복잡도 / 공간복잡도

- 시간복잡도: O(N²)
- 공간복잡도: O(N)

## 문제 4

### 문제 정보

- 문제명: 자물쇠와 열쇠
- 문제 링크: https://school.programmers.co.kr/learn/courses/30/lessons/60059
- 사용한 알고리즘 / 자료구조: 구현, 완전탐색

### 처음 접근 방식

문제의 조건을 우선 정리했다.


- `0`은 홈, `1`은 돌기
- 열쇠는 이동할 수 있다.
- 열쇠는 90도 단위로 회전할 수 있다.
- 자물쇠의 모든 홈이 열쇠의 돌기로 채워져야 한다.
- 자물쇠의 돌기와 열쇠의 돌기가 겹치면 안 된다.
- 열쇠의 일부는 자물쇠 영역 밖으로 벗어날 수 있다.

따라서 열쇠를 0도, 90도, 180도, 270도로 회전시키고, 가능한 모든 위치에 올려보면서 자물쇠가 열리는지 확인하는 완전탐색 문제라고 생각했다.

### 풀이 방법

이 문제의 핵심은 열쇠가 자물쇠 밖으로 벗어날 수 있다는 것이었다.

처음에는 lock 배열 안에서만 key를 이동시키려고 했지만, 이렇게 하면 key의 일부만 lock과 겹치는 경우를 표현하기 어렵다.  

그래서 lock을 더 큰 배열의 중앙에 배치하는 방식으로 문제를 해결할 수 있다.

확장 배열의 크기는 다음과 같이 잡을 수 있다.

```text
expandedSize = lock.length + 2 * (key.length - 1)
```

이렇게 하면 자물쇠의 상하좌우에 key.length - 1만큼의 여유 공간이 생긴다.
따라서 key가 lock의 가장자리만 살짝 겹치는 경우까지 모두 표현할 수 있다.

풀이 흐름은 다음과 같다.

1. 기존 lock을 확장된 배열의 중앙에 복사한다.
2. key를 0도, 90도, 180도, 270도로 회전한다.
3. 각 회전 상태마다 key를 확장 배열의 모든 가능한 위치에 올려본다.
4. key 값을 확장 배열에 더한다.
5. lock이 위치한 중앙 영역이 모두 1인지 확인한다.
6. 중앙 영역이 모두 1이면 자물쇠가 열린 것으로 판단한다.
7. 확인이 끝나면 key를 다시 제거하거나, 매번 복사 배열을 만들어 다음 경우를 검사한다.

여기서 중앙 lock 영역이 모두 1이어야 하는 이유는 다음과 같다.

- lock의 홈 0에 key의 돌기 1이 들어가면 1이 된다.
- lock의 돌기 1에 key의 빈칸 0이 겹치면 그대로 1이다.
- lock의 홈 0이 그대로 남으면 0이므로 실패다.
- lock의 돌기 1에 key의 돌기 1이 겹치면 2가 되므로 실패다.

즉, key를 올려본 뒤 lock 영역의 모든 값이 정확히 1이면 성공이다.


### 어려웠던 지점

가장 어려웠던 부분은 key의 이동 범위를 어떻게 잡을지였다.

문제 조건상 key는 lock 밖으로 벗어날 수 있는데, 단순히 lock 배열 안에서만 key를 이동시키면 key가 lock의 일부에만 걸쳐지는 경우를 처리하기 어렵다.

그래서 lock을 확장해야 한다는 방향은 떠올렸지만, 확장 배열의 크기를 어느 정도로 잡아야 하는지, 그리고 기존 lock 값을 확장 배열의 어느 위치에 넣어야 하는지가 어려웠다.

또한 key를 특정 위치에 올렸을 때, 실제로 자물쇠가 열렸는지 확인하는 기준도 처음에는 명확하지 않았다.

하지만 key 값을 더한 뒤 중앙 lock 영역이 모두 1인지 확인하면 된다는 방식으로 생각하면 비교 로직을 단순화할 수 있다.

### 해결 방법

lock 주변에 key가 벗어날 수 있는 여유 공간을 만들기 위해 확장 배열을 사용한다.

lock의 크기를 N, key의 크기를 M이라고 하면 확장 배열의 크기는 다음과 같다.

N + 2 * (M - 1)

그리고 기존 lock은 확장 배열의 중앙에 배치한다.

```text
시작 위치 = M - 1
끝 위치 = M - 1 + N - 1
```


이후 key를 회전시키며 모든 위치에 올려본다.

key를 올려볼 때는 확장 배열의 복사본을 만들고, 해당 위치에 key 값을 더한다.
그 다음 중앙에 있는 lock 영역만 검사한다.

```text
중앙 lock 영역의 모든 값이 1이면 성공
하나라도 0 또는 2가 있으면 실패
```


이 방식으로 풀면 key가 lock 밖으로 나가는 경우도 확장 배열 안에서 자연스럽게 표현할 수 있다.


### 시간복잡도 / 공간복잡도

- 시간복잡도: O(4 * (N + M)^2 * (M^2 + N^2)) -> lock 크기를 N, key의 크기를 M이라고 하면, 확장 배열의 크기는 대략 N + 2M인데, key는 총 4번 회전할 수 있고, 각회전마다 key를 확장 배열의 가능한 위치에 올려봄. -> 각 위치마다 key를 더하는 데 O(M^2). lock 영역 검사하는 데 O(N^2) 걸림
- 공간복잡도: O((N + M)^2) -> 확장 배열 사용해서

### 새롭게 알게 된 점

회전 자체보다 이동 범위를 어떻게 모델링할지가 더 중요한 구현 문제였다.

아래 구현이 중요했다.

- key 회전 구현
- lock을 확장한 배열 구성
- key를 모든 위치에 올려본 뒤 중앙 lock 영역이 모두 1인지 확인
